1) Lessons learned (cose da tenere a mente quando fai add-on Blender “seri”)
A. Import e “boot” dell’add-on: tutto deve essere ultra-robusto

Blender abilita un add-on facendo import dell’intero package. Se un singolo import fallisce, tutto l’add-on non parte.

Mai lasciare import fragili tipo from X import Y se Y può cambiare nome durante refactor.

Se stai rinominando funzioni pubbliche, usa una strategia:

alias/wrapper di compatibilità (es: ensure_neighbor_color_nodes che chiama ensure_floretion_neighbor_material o viceversa)

oppure aggiorni tutti gli import chiamanti nello stesso commit.

Regola d’oro: l’add-on deve poter essere importato anche se l’utente non ha ancora creato nulla nella scena.

B. Blender 5 / Python 3.11: type hints possono rompere l’import

Quello che ti ha morso con NameError: Floretion is not defined è un classico.

Se scrivi type hints con classi non importate (Floretion) e le annotation vengono valutate → boom.

Soluzioni sicure:

from __future__ import annotations in cima al file

from typing import TYPE_CHECKING + import solo sotto if TYPE_CHECKING:

oppure usare stringhe "Floretion" (ma meglio insieme a 1)

Lesson: in add-on Blender, i type hints sono utili, ma vanno “sandboxati” per non bloccare l’import.

C. Firma delle funzioni: keyword-only vs posizionali (compatibilità chiamanti)

Hai visto _ensure_viewport_label() che “prende 0 args ma ne riceve 5”.

Quando cambi una firma (es. rendi keyword-only), rompi subito i call-site.

Buona pratica: se è una funzione usata da più moduli, metti un wrapper compatibile:

accetta *args, **kwargs

normalizza internamente

così puoi refactorare senza dover aggiornare 10 file in lockstep.

D. Attributi mesh: scrivili nel posto giusto (BMesh vs Mesh)

Per roba tipo CORNER color e FACE float ci sono 2 pipeline:

BMesh → to_mesh: scrivere layer su BMesh prima di bm.to_mesh(me) è spesso il modo più “naturale”.

Mesh attributes: post conversione puoi usare me.attributes.new(... domain='FACE') e me.color_attributes.new(... domain='CORNER').

Lesson: scegli UNA via principale e una di fallback.
E soprattutto: evita di “mescolare” scrittura e lettura in punti sparsi (il “mega-mess” che descrivevi).

E. Shader nodes: separa “dati” da “visualizzazione”

Una cosa che ti ha creato confusione era:

colori calcolati “normali”

colori “neighbors”

e poi override “in build_object_from_floretion” che faceva casino.

Lesson pulito:

floretion_color = colore base (ABS_HSV ecc.)

color_edges / color_verts / color_edges_and_verts = colori neighbor

lo shader sceglie quale leggere (switchando attribute_name), invece di riscrivere tutto.

F. UI toggles: devono fare qualcosa anche senza rebuild

Le checkbox “show curve/centroids” che “non fanno nulla” succede quando:

esiste la property UI

ma non c’è update handler o apply function che setta hide_viewport/hide_render.

Lesson: per roba di visibilità, meglio fare:

update callback → chiama apply_helper_visibility()

e opzionale rebuild solo se serve davvero.

G. “Non perdere feature” durante refactor: l’API pubblica è sacra

Nel tuo caso, ui_panels.py chiamava "TRI" e tu avevi ristretto l’enum → pannello rotto.

Lesson: tutto ciò che il pannello chiama (Operator.action, props, idname) è “API pubblica” e va mantenuta o aliasata.

2) I socket “Vector / Factor / Alpha” nel nodo Attribute: cosa sono e come si usano?

Il nodo Attribute (ShaderNodeAttribute) in Blender legge un attributo per shading. A seconda del tipo di attributo, espone output diversi:

Color: output RGBA (in pratica un Vector RGB + alpha)

Vector: output XYZ (tipico per attributi vettoriali: normal custom, direction, tangent, velocity, ecc.)

Factor: output float singolo (per attributi scalari: maschere, pesi, intensità)

Alpha: canale alpha separato (se l’attributo è un color con alpha o se Blender lo ricava come quarto canale)

Nel tuo caso (Color Attributes CORNER)

floretion_color, color_edges, ecc. sono Color Attributes (dominio CORNER).

Quindi:

Color = il vero output che ti interessa per Base Color / Emission.

Alpha = 1.0 quasi sempre (a meno che tu scriva alpha diverso nei loop colors).

Factor spesso è una conversione “luminanza”/media o un fallback (dipende da Blender), utile se vuoi una maschera da un colore.

Vector per un color di solito coincide con RGB (senza alpha), ma in alcuni casi è “solo” una vista come vettore.

Esempi pratici d’uso (super utili)

Vuoi che i neighbors influenzino solo la roughness?
Usa Factor (o separa l’RGB e fai una media) → Roughness.

Vuoi usare il canale alpha come maschera per mixare due shader?
Usa Alpha → Factor di un Mix Shader.

Vuoi trattare RGB come vettore e farci dot product con una direzione (effetto “iridescenza”/highlight)?
Usa Vector → Vector Math.

Sintesi: per “colorazione visibile” di facce, quasi sempre userai Color (e magari Alpha come maschera). Gli altri output sono lì perché il nodo è generico e può leggere attributi non-color.
